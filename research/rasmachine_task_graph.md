## Goal

Build the foundational interface for rescoring INDRA Statement corpora through our composed scorer. Rasmachine (`latest_statements_rasmachine.json`, ~8,724 stmts) is the first concrete corpus. The interface must equally handle:

- **Truth-bearing corpora** — our 482-record gold pool, INDRA's published belief, source-DB curation flags. Validity surface: precision / recall / calibration / per-step confusion.
- **No-truth corpora** — rasmachine, future imports we don't own labels for. Validity surface: INDRA-belief calibration, inter-evidence consistency, gold-overlap agreement.

Same persistence, same viewer, same validity dashboard — branching only on what truth signals are present for the loaded slice. Persistence must survive scorer-architecture iterations (content-addressed, append-only, version-tagged). Presentation must hold a Bret Victor / Tufte register: explorable, reactive, dense without clutter — controlled chaos at scale.

Downstream: re-emit beliefs in INDRA-native JSON so the EMMAA collaborator can rebuild the rasmachine model and measure lift; same export path serves any future corpus.

## Truth-bearing structures we inherit

INDRA's `Statement` carries non-trivial truth-adjacent signal that the schema must preserve, not invent around. Confirmed by 1.1 catalog (2026-05-09):

- `Statement.belief` — INDRA's published belief score ∈ [0,1]. Calibration anchor for no-truth corpora.
- `Statement.get_hash(shallow=True)` — 14-nibble (56-bit) content-stable identity; canonical `stmt_hash`. `shallow=False` (16-nibble, evidence-inclusive) → `extraction_hash` for tracking specific dumps.
- `Statement.supports` / `supported_by` — **stored as UUID lists in JSON** (NOT live object refs); graph reconstruction is a separate operation, modeled as `supports_edge` table, not a column.
- `Evidence.source_hash` (content-addressed; primary key) + `Evidence.stmt_tag` (parent's full_hash; FK link).
- **`Evidence.epistemics` is not loose annotation — it is a control gate.** Carries `direct/indirect`, `negated`, `curated` flags from upstream curators. **These auto-register as `truth_label` rows on ingest** under truth_set `indra_epistemics` — every INDRA dump gives free truth signal.
- `Agent.db_refs` namespaces (FPLX, UPPRO, HGNC, UP, CHEBI, GO, MESH, MIRBASE, DOID, HP, EFO) — grounding truth.
- `Agent.matches_key()` = `entity_matches_key()` + `state_matches_key()` — canonical agent identity. `entity_matches_key()` alone for "same protein, ignoring state" queries.
- Statement subclasses (Phosphorylation, Activation, Inhibition, Complex, Translocation, Conversion, RegulateAmount, …) carry type-specific fields (`enz`/`sub`/`residue`/`position` for modifications; `subj`/`obj`/`obj_activity`/`is_activation` for RegulateActivity). Discriminator is the `type` field. **Schema keeps `raw_json` per statement; projects key fields where queryable, doesn't try to flatten subclasses.**

Our project-side truth: 482-record gold pool + U2 holdout per-probe gold.
Source-DB truth (when ingested): BioGRID, Pathway Commons, SIGNOR curation flags via `source_api` + `epistemics.curated`.

## Glossary

- **scorer_step**: a row in the `scorer_step` table; one of `step_kind ∈ {parse_claim, build_context, substrate_route, subject_role_probe, object_role_probe, relation_axis_probe, scope_probe, grounding, adjudicate, aggregate}`. Currently `parse_claim`, `build_context`, `substrate_route`, the 4 substrate-answered probes, and `aggregate` are emitted (`grounding`/`adjudicate` decomposition deferred to user's orchestrator code).
- **truth_set**: a registered source of labels (gold pool, INDRA published belief, source-DB curation flag, …). Plural per corpus.
- **truth_label**: `(truth_set, target_kind, target_id, [relation_target_id], field, value_text|value_json) → labeled fact`. Polymorphic over stmt/evidence/agent/scorer_step targets; supports edge labels and non-scalar values.
- **belief delta**: `our_belief − INDRA_published_belief`, per statement.
- **stmt_hash**: `Statement.get_hash()` — INDRA's canonical statement identity.
- **scorer_version**: git commit hash of indra-belief at scoring time.
- **model_id**: identifier of the LLM used per scorer_step (captured for reproducibility).
- **lossless loader**: every field that survives INDRA's `Statement.to_json()` reaches DuckDB cleanly.
- **controlled chaos** (operational, D10): every dense view exposes ≥1 reactive parameter and ≤2 chrome elements; per-step grammars are explorable (interaction → downstream change), not just legible.
